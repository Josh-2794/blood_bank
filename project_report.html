1. Project Overview

LifeFlow is a full-stack Blood Bank and Donation Management System built as a DBMS project. It connects blood donors with patients in need, provides real-time inventory tracking, and offers an admin dashboard for management.

AttributeDetails
Project NameLifeFlow Blood Bank & Donation Management System
TypeWeb Application (Single Page App + PHP REST API)
FrontendHTML5, CSS3, Vanilla JavaScript (ES6+)
BackendPHP 8.x (REST API)
DatabaseMySQL 8.x via MySQLi
ServerXAMPP (Apache + MySQL)
Database Nameblood_bank_db

Core Features

  • Donor registration with blood group, city, availability status
  • Blood donor search by blood group, city and availability
  • Blood request submission (patients/hospitals)
  • Real-time blood inventory display with stock percentage
  • Admin dashboard — manage donors, approve/reject requests, update inventory
  • Donation log tracking with stored procedures
  • Admin authentication with hashed passwords (bcrypt)

4.3 Views

Views are virtual tables that simplify complex queries.

View 1: v_critical_blood

Returns all blood types sorted by stock level with status labels (Critical / Low / OK).

CREATE VIEW v_critical_blood AS
SELECT blood_group, units, max_units,
       ROUND(units/max_units * 100, 1) AS stock_pct,
       CASE
           WHEN units <= 10 THEN 'Critical'
           WHEN units <= 20 THEN 'Low'
           ELSE 'OK'
       END AS status
FROM blood_inventory
ORDER BY units ASC;

Used by: api/inventory.php GET — powers the inventory cards on the Inventory page.

View 2: v_pending_requests

Returns all pending blood requests sorted by urgency (days remaining).

CREATE VIEW v_pending_requests AS
SELECT id, patient_name, blood_group, units_required,
       hospital, required_by,
       DATEDIFF(required_by, CURDATE()) AS days_left
FROM blood_requests
WHERE status = 'Pending'
ORDER BY required_by ASC;

Used by: Admin dashboard to prioritise urgent requests.

4.4 Stored Procedures

Stored procedures are pre-compiled SQL blocks that run multiple statements atomically.

Procedure 1: add_donation()

Called when a donation is logged. It inserts the donation record AND updates inventory in one call.

CALL add_donation(donor_id, blood_group, units, date);

-- What it does internally:
-- 1. INSERT into donations table
-- 2. UPDATE blood_inventory SET units = units + p_units
-- 3. UPDATE donors SET last_donation = p_date

Procedure 2: fulfill_request()

Called when admin marks a request as Fulfilled. Deducts units from inventory.

CALL fulfill_request(request_id);

-- What it does internally:
-- 1. SELECT blood_group & units_required from blood_requests
-- 2. UPDATE blood_inventory SET units = GREATEST(0, units - v_units)
-- 3. UPDATE blood_requests SET status = 'Fulfilled'
Why use stored procedures? They group related SQL statements, reduce network round-trips, and prevent partial updates — if one statement fails, the whole procedure can be rolled back.

4.5 Normalization

First Normal Form (1NF) ✓

  • All columns hold atomic (single) values — no comma-separated blood groups or multi-valued fields
  • Every table has a primary key (id)
  • No repeating groups

Second Normal Form (2NF) ✓

  • All tables use a single-column primary key, so there can be no partial dependency
  • Every non-key column depends fully on the primary key

Third Normal Form (3NF) ✓

  • No transitive dependencies — donor details are in donors, inventory in blood_inventory, etc.
  • Blood group stock is not stored inside donors — it lives in its own blood_inventory table
Note on blood_group redundancy: The blood_group column appears in donors, donations, and blood_requests. This is intentional — it avoids costly joins and is a controlled denormalization acceptable in this design.

9. Expected DBMS Lecturer Questions & Answers

These are the most commonly asked questions in DBMS project vivas. Study these carefully — the lecturer will likely pick from this list.

— Database Design —

What is the primary key in each table and why?
Every table has an id INT AUTO_INCREMENT PRIMARY KEY. This is a surrogate key — it uniquely identifies each row, is never null, never changes, and is efficient for joins. For blood_inventory, blood_group is also UNIQUE since only one row per blood type should exist.
What is a foreign key? Where is it used in your project?
A foreign key is a column that references the primary key of another table, enforcing referential integrity. In this project, donations.donor_id is a foreign key referencing donors.id. With ON DELETE CASCADE, if a donor is deleted, all their donation records are automatically deleted too.
What is normalization? Which normal form is your database in?
Normalization is the process of organizing a database to reduce redundancy and improve data integrity. This project is in Third Normal Form (3NF): (1NF) all columns hold atomic values, no repeating groups; (2NF) all non-key attributes depend on the full primary key; (3NF) no transitive dependencies — e.g., blood inventory is in its own table, not inside the donors table.
Why is there a separate blood_inventory table instead of just counting donor blood groups?
Counting donor blood groups would give the number of registered donors, not the actual units of blood available. The blood_inventory table tracks the physical blood stock — which increases when donations are processed and decreases when requests are fulfilled. These are fundamentally different things.
What is an ENUM type? Why did you use it for blood_group and status?
ENUM is a MySQL data type that restricts a column to a predefined list of values. It saves storage compared to VARCHAR and enforces data integrity at the database level — for example, you cannot accidentally insert 'X+' as a blood group. It makes queries faster too since MySQL stores ENUMs as integers internally.

— SQL Queries —

Write a query to find all available donors with blood group O+.
SELECT first_name, last_name, phone, city
FROM donors
WHERE blood_group = 'O+' AND availability != 'Not Available' AND is_active = 1;
Write a query to find which blood groups are critically low (less than 10 units).
SELECT blood_group, units, max_units
FROM blood_inventory
WHERE units < 10
ORDER BY units ASC;
Write a query to count total donations per blood group.
SELECT blood_group, SUM(units) AS total_units, COUNT(*) AS total_donations
FROM donations
GROUP BY blood_group
ORDER BY total_units DESC;
Write a query to get donors who have never donated.
SELECT id, first_name, last_name, blood_group
FROM donors
WHERE last_donation IS NULL AND is_active = 1;
How do you join the donors and donations tables?
SELECT d.first_name, d.last_name, d.blood_group,
       dn.donated_on, dn.units
FROM donors d
INNER JOIN donations dn ON d.id = dn.donor_id
ORDER BY dn.donated_on DESC;
An INNER JOIN returns only donors who have at least one donation. Use LEFT JOIN to include donors with no donations.

— Views —

What is a View? Why did you create views in your project?
A view is a saved SQL query stored as a virtual table. It doesn't store data — it generates results dynamically each time it's queried. I created v_critical_blood to simplify getting inventory status without repeating the CASE logic in every query, and v_pending_requests to easily see urgent pending requests sorted by deadline. Views improve code reuse and hide query complexity.
What is the difference between a View and a Table?
A table physically stores data on disk. A view is a virtual table — it stores a query definition, not data. Every time you query a view, MySQL runs the underlying SELECT. Views cannot (usually) be inserted into directly. They are useful for read-only reporting queries.

— Stored Procedures —

What is a Stored Procedure? Explain the ones in your project.
A stored procedure is a named block of SQL code saved in the database that can be called with CALL. In this project: add_donation() — when a donation is logged, it inserts into the donations table, updates the donor's last donation date, and increments blood_inventory units in one atomic call. fulfill_request() — when a request is fulfilled, it decrements inventory and updates the request status.
Why use a stored procedure instead of running the SQL from PHP?
Three reasons: (1) Atomicity — all statements in the procedure succeed or fail together, preventing partial updates. (2) Performance — stored procedures are pre-compiled by MySQL, faster than sending raw SQL each time. (3) Security — the PHP app only needs EXECUTE permission on the procedure, not direct UPDATE/INSERT access to tables.

— Security & Best Practices —

How did you prevent SQL Injection in your project?
All database queries use MySQLi prepared statements with bind_param(). The user input is passed as a parameter — it is never concatenated directly into the SQL string. Additionally, all inputs pass through the clean() function which strips HTML tags and escapes special characters.
How are passwords stored in the database?
Admin passwords are hashed using bcrypt via PHP's password_hash() function. The hash is stored in the database, not the plain text. On login, password_verify() compares the entered password against the stored hash. Even if the database is compromised, the real passwords remain safe.
What does the is_active column in donors do?
This is a soft delete flag. Instead of physically deleting a donor record (which would break the donation history foreign keys), we set is_active = 0. All queries filter with WHERE is_active = 1, so soft-deleted donors are hidden but their donation records are preserved for historical reporting.

— Architecture & Design —

What is a REST API? How does your PHP backend follow REST principles?
REST (Representational State Transfer) is an architectural style where resources are accessed via URLs and standard HTTP methods. In this project: GET fetches data, POST creates new records, PUT updates existing records. Each resource has its own endpoint (donors.php, requests.php, inventory.php). All responses are JSON. The API is stateless — each request carries all needed information.
What is a Single Page Application (SPA)?
An SPA loads one HTML page and dynamically shows/hides content sections using JavaScript, instead of loading a new page from the server on each navigation. In LifeFlow, showPage('donate') hides all other page divs and shows #page-donate — no server request is made. This gives a faster, app-like user experience.
What is the 3-tier architecture and how does your project use it?
3-tier architecture separates an application into: Presentation tier (HTML/CSS/JS — what the user sees), Application tier (PHP API — business logic, validation, query building), and Data tier (MySQL — data storage, views, procedures). Each tier communicates only with the adjacent tier — JavaScript never touches MySQL directly; it always goes through PHP.
What is ACID in databases? Does your project handle it?
ACID stands for: Atomicity (all or nothing), Consistency (data remains valid), Isolation (concurrent transactions don't interfere), Durability (committed data persists). The stored procedures in this project handle atomicity — add_donation() ensures the donation log and inventory update both happen together. MySQL's InnoDB engine provides full ACID compliance by default.
What would you improve in this project given more time?
Good answers: (1) Add database transactions (BEGIN / COMMIT / ROLLBACK) around stored procedures. (2) Add role-based access control — only authenticated admins can access admin API endpoints. (3) Add indexing on frequently searched columns like blood_group and city. (4) Implement pagination for large donor lists. (5) Add email/SMS notification when matching donors exist for a new request.
